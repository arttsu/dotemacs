* Overview

This document tracks the migration of the GTD (Getting Things Done) system from config.org.bkp to the current configuration. The migration follows a phased approach to ensure testability and minimize risk.

* Naming Convention Changes

** OLD → NEW Mapping
- =personal= → =local= (context-specific content)
- =open= → =shared= (cross-context content)

** Directory Structure
- =~/org-personal/gtd/= → =~/org-local/gtd/=
- =~/org-open/gtd/= → =~/org-shared/gtd/=

** Variable Naming Examples
- =my-gtd-personal-dir= → =my-gtd-local-dir=
- =my-gtd-open-projects= → =my-gtd-shared-projects=
- =my-gtd-personal-inbox= → =my-gtd-local-inbox=

* Notes & Decisions

** Architectural Decisions
- Use "local" and "shared" instead of "personal" and "open" for clarity
- Maintain phased approach for testability
- Keep template files in =capture-templates/= subdirectory
- Preserve all existing functionality while updating naming

** Testing Strategy
- Each phase must be testable independently
- Can stop at any phase if issues arise
- Manual testing of core workflows at each phase
- Verify no regressions in existing org functionality

* Future Improvements

** Cross-linking System
The cross-linking functionality from Phase 5 would provide enhanced traceability between projects and areas:

*** Function: =my-gtd-copy-heading-to-area=
- **Purpose**: Copy project tasks to their linked area files while maintaining bidirectional traceability
- **Key Binding**: =C-c o c=
- **Features**:
  - Smart area detection through directory mapping and AREA links
  - Bidirectional linking with =COPIED_FROM= properties
  - Prevention of duplicate copies with =:COPIED:= tags
  - Integration with multi-context GTD setup (local/shared)

*** Benefits of Future Implementation
1. **Enhanced Traceability**: Track items from areas back to originating projects
2. **Context Preservation**: Maintain project context while enabling area-based management
3. **Workflow Continuity**: Allow work to flow naturally between project and area contexts
4. **Robust Linking**: Uses org-id system for links that survive file reorganization

*** Implementation Notes
- Would require updating variable names from personal/open to local/shared
- Should integrate with existing GTD directory structure
- Could leverage org-node for enhanced cross-linking capabilities
- Might benefit from modern Emacs completion frameworks
